# Production Deployment Recommendations

For a production deployment, the first priority is replacing the file-based word index with a purpose-built search backend such as Elasticsearch or Apache Solr. The current file-per-letter approach works for demonstration but does not scale to billions of indexed terms — a proper inverted index with TF-IDF or BM25 ranking would dramatically improve both search quality and performance. The crawler's MySQL-backed queue should be migrated to a dedicated distributed task queue (e.g., Celery with Redis or RabbitMQ) to enable horizontal scaling across multiple worker nodes. Each worker would pull URLs from the shared queue, respecting per-domain politeness rules (robots.txt, crawl-delay) and global rate limits enforced via a centralized token bucket. DNS resolution caching, persistent HTTP connection pools, and a headless browser fallback for JavaScript-rendered pages would further improve crawl throughput and coverage.

On the operational side, the system needs comprehensive observability: structured logging shipped to a centralized platform (ELK or Datadog), Prometheus metrics for crawl rate / queue depth / error ratios, and alerting on back-pressure events or elevated failure rates. Data durability should be handled by periodic snapshots of the index and WAL-based replication for the metadata database. Security hardening includes sandboxing the fetch layer to prevent SSRF, enforcing request-size limits, and stripping or sanitizing any stored content to avoid XSS when displaying results. Finally, deploying the crawler and search tiers independently — containers orchestrated by Kubernetes with autoscaling policies — allows each component to scale according to its own resource profile: CPU-bound crawling vs. I/O-bound search serving.
